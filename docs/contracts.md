# Livo 协作契约

## 分层边界

- `src/main` 负责持久化、网络抓取、系统集成、后台任务和 AI provider 调用。
- `src/preload` 只暴露渲染层需要的受限 API，不透传任意 Node 或 Electron 能力。
- `src/renderer` 负责 UI 状态和交互编排，不直接访问文件系统、数据库或系统 API。
- `src/web` 只能依赖浏览器能力和 Web 适配层，不引用 Electron 专属 API。
- `src/shared` 只放平台无关的类型、常量、设置归一化、规则和纯工具函数。

## IPC 契约

- IPC channel 统一在共享类型或集中位置声明，避免在调用点散落硬编码字符串。
- 主进程 handler 返回结构化结果，失败时返回明确错误信息或抛出不可恢复错误。
- 渲染层发起的异步请求需要具备过期保护；流式事件必须携带可校验的 `requestId`。
- preload API 是渲染层唯一入口。新增主进程能力时先设计 preload 的最小暴露面。

## 数据契约

- 数据写入统一经过主进程数据层，渲染层不构造持久化副作用。
- 入库前完成订阅源归一化、条目去重和规则评估，UI 层只消费已归一化的视图数据。
- 设置读取后必须经过 `normalizeSettings`，写入时使用合并语义，避免部分字段缺失破坏默认值。
- `shared` 中的视图模型映射保持纯函数，不能读取运行时环境或触发 I/O。

## AI 契约

- AI provider 配置由主进程读取并校验，渲染层不直接调用外部模型服务。
- prompt 默认值与回退逻辑集中在 AI prompt/service 层，调用点只传业务输入。
- 流式摘要、翻译、对话事件必须按 `requestId` 过滤，过期响应不能回写当前页面。
- 用户修改 AI key、base URL、model 或 prompt 后，正在进行的渲染层会话应停止接收旧配置结果。

## UI 状态契约

- 页面级 Hook 负责组合业务状态，展示组件保持可复用和可测试。
- 用户可恢复的错误在当前交互位置展示，并提供局部重试入口。
- 设置页避免把临时输入直接扩散到无关状态；需要持久化时通过 settings store 统一写入。
- 列表、阅读器和浮层应共享同一套可复用渲染组件，除非交互模型确实不同。

## 文档契约

- 涉及跨层边界、持久化格式、IPC 形状或后台任务策略的改动，需要同步更新架构或契约文档。
- 大型功能设计放在 `docs/plans/`，并记录目标、非目标、数据流、异常路径和验证方式。
